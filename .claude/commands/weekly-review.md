# /weekly-review コマンド

週次の振り返りを行い、inbox の棚卸し、未整理ノートの分類、今週のサマリーを生成します。

## 入力
- `$ARGUMENTS`: オプションでプロジェクト名を指定（例: `--project churn-reduction-2025`）
- 指定がなければ、最新のプロジェクトを対象にする

## 処理手順

### Step 1: inbox の棚卸し

`knowledge/inbox/` を確認し、以下に分類してください:

| 分類 | 説明 |
|------|------|
| **Distill候補** | アトミックノート化すべき議事録 |
| **アーカイブ候補** | 既にDistill済み、または不要 |
| **保留** | 判断保留（後日確認） |

出力形式:
```markdown
## inbox 棚卸し

### Distill候補（要処理）
- [ ] 2025-12-XX_meeting.md - 理由: まだノート化されていない

### アーカイブ候補
- [x] 2025-12-01_kickoff_meeting.md - 理由: 主要ノート作成済み

### 保留
- [ ] 2025-12-YY_memo.md - 理由: 内容確認が必要
```

### Step 2: 今週の意思決定を抽出

`knowledge/notes/note-decision-*` から、今週作成・更新されたものを抽出:

```markdown
## 今週の意思決定

| 日付 | 決定事項 | ノート |
|------|---------|--------|
| 12/22 | 追加予算承認 | [[note-decision-budget-approval]] |
```

### Step 3: 未決事項の確認

`knowledge/notes/note-open-*` から、期限が近いものを抽出:

```markdown
## 未決事項（期限注意）

| 項目 | 期限 | 残り日数 | 担当 |
|------|------|---------|------|
| RecoTech契約 | 12/25 | **3日** | 田中 |
```

### Step 4: リスクの確認

`knowledge/notes/note-risk-*` から、対応が必要なものを抽出:

```markdown
## 要注意リスク

| リスク | ステータス | 次アクション |
|--------|-----------|-------------|
| スケジュール遅延 | 監視中 | 週次で確認継続 |
```

### Step 5: 次週のアクション

`projects/{project}/README.md` の「次のアクション」を確認し、更新提案を作成:

```markdown
## 次週のアクション

| アクション | 担当 | 期限 |
|-----------|------|------|
| RecoTech契約締結 | 田中 | 12/25 |
| オンボーディング本番反映 | 佐藤 | 12/28 |
```

### Step 6: プロジェクトREADMEの更新提案

`projects/{project}/README.md` の「今週の状況」セクションを更新する提案を生成:

```markdown
## README更新提案

### 今週の状況（12/29更新）

#### サマリー
- **Phase1**: 完了 ✅
- **Phase2**: 準備中

#### 今週の成果
1. オンボーディング改善A/Bテスト結果確定: +4.6pt
2. 経営会議で追加予算350万円承認

#### 今週の課題
1. RecoTech社との契約条件交渉（12/25期限）

---
この内容で `projects/churn-reduction-2025/README.md` を更新してよいですか？
```

## 出力形式

```markdown
# 週次レビュー（YYYY-MM-DD）

## 1. inbox 棚卸し
...

## 2. 今週の意思決定
...

## 3. 未決事項（期限注意）
...

## 4. 要注意リスク
...

## 5. 次週のアクション
...

## 6. README更新提案
...

---
**処理候補**:
- `/distill 2025-12-XX_meeting.md` を実行しますか？
- `projects/.../README.md` を更新しますか？
```
